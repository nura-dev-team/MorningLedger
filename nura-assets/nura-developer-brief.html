Section 01

What You're Building

NURA is a real-time financial operating system for hospitality operators. Restaurants run on 3–5% margins. Prime cost — food, beverage, and labor — eats 60–65% of every dollar. Operators don't see those numbers until month-end, 30 days after the money is already gone.

The existing tools fall into two buckets: backward-looking reporting tools (MarginEdge, xtraCHEF) that explain the past, and complex enterprise platforms (Restaurant365) built for finance teams. Neither is designed for an operator on the floor making a real decision at 9am on a Tuesday.

NURA is the third option — a real-time decision surface. An operator opens it and immediately knows: what's been spent, what's left, what's driving risk, and what they can still do about it today. The product is built around one behavioral loop:

See → Decide → Move on. That's the entire product philosophy. Every screen, every data point, every interaction must serve that loop. If it doesn't help the operator make a faster, better decision, it doesn't belong.

The Context You Need to Know

Keem (the founder) is Corporate Director of Culinary & Concept Design at Donohoe Hospitality, a 22-property portfolio. He has been running NURA manually — as a checkbook spreadsheet — across real properties with real invoices, real vendors, and real POS data. The first pilot property is SYN, a live Donohoe restaurant.

This means you are not building in a vacuum. You have real data, real GL codes, real vendor names, and a real operator who will use what you build within weeks. That's an enormous advantage. Use it.

Section 02

The Product — 6 Screens Locked

These six screens are not up for debate. They have been designed, validated against real operator behavior, and locked. Every screen answers a specific question. That question is the north star for what data it shows.

1. Home Priority 1
"Where do I stand today?"
  • Prime Cost MTD — the single most important number
  • Sales MTD with weekly breakdown
  • Remaining food budget at a glance
  • COGS spend vs budget as a progress bar
  • Status narrative — plain English interpretation of the numbers
  • Recent transactions (last 4–5 invoices)
  • What the operator can still influence this week
2. Prime Cost Priority 1
"What's driving health or risk?"
  • Prime cost % MTD — large and dominant
  • Breakdown: F&B % + Labor % stacked visually
  • Weekly revenue trend showing whether sales are building
  • "What's driving this" — plain-language bullets
  • "What you can still influence" — actionable bullets
  • Target % shown clearly (62% for SYN)
3. Budgets Priority 1
"What do I have left — and why?"
  • Every GL category: actual spend, monthly budget, remaining, utilization %
  • Progress bars — green under budget, orange over budget
  • One-line status summary at top
  • Over-budget categories highlighted with context
  • SYN categories: Food, Liquor, Wine, Beer, Operating Supplies, Uniforms
4. Acknowledge & Code (Approvals) Priority 2
"Should I approve this right now?"
  • Pending invoices as cards: vendor, amount, category, budget impact
  • Budget context: remaining before AND after approval
  • Sales pace context included on each card
  • Within budget: one-tap Approve button
  • Over budget: explanation dropdown required before approval
  • Reason options: Recurring contract / One-time exception / Vendor issue / Custom note
  • Hold option on every card
5. Ledger + Master Vendor List Priority 2
"Is this clean and explainable?"
  • Full invoice ledger: date, vendor, description, amount, GL code pill, status
  • Summary stats: invoices coded, total spend, audit variance
  • Master Vendor List sub-tab: vendors, email connection status, AP contact, delivery frequency, gap alerts
  • Month-end should feel like confirmation, not discovery
6. Controller Dashboard Phase 2
"How are all my properties performing?"
  • Portfolio view: status signal per property (green/amber/orange/red)
  • Prime cost MTD with trend arrow per property
  • Budget authority: controller can adjust budgets by revenue pace
  • Month-end export: one-click GL reports per property (saves 8–12 hrs/client/month)
  • Build this after the first 5 screens are solid

Section 04

Core Workflows to Implement

Workflow 1 — Invoice Ingestion (The Central Loop)

This is the most important workflow in the product. One action, five outcomes. A single invoice upload simultaneously triggers all five steps below.

"One action, five outcomes" — A single invoice upload triggers: (1) read, (2) GL code, (3) dedup check, (4) budget impact calculation, (5) route to result. The operator does one thing. The system does five.
1
Read the Invoice
Extract vendor, invoice number, date, line items, amounts, totals. For v1, simulate with a manual upload form — don't overcomplicate OCR yet.
2
Assign GL Code
Auto-assign from vendor default mapping. For v1, use layer (a) only: vendor default. Layers (b) description keyword and (c) operator preference rules come later.
3
Dedup Check
Check invoice number + vendor + amount combination. If match found → flag as duplicate, do not process. This protects against double-entry.
4
Calculate Budget Impact
Show: remaining before, remaining after, over/under status. This determines whether approval requires a reason.
5
Route to Result
Within budget → one-tap approve. Over budget → explanation required. After approval: ledger, budgets, prime cost, and Home screen all refresh immediately.

Workflow 2 — Approval Flow Logic

ScenarioUI BehaviorRequired Action

Workflow 3 — Budget Calculation Logic

// Core budget calculation
remaining = monthly_budget - sum(approved_invoices_for_gl_code)

// Utilization %
utilization = (sum(approved_invoices) / monthly_budget) * 100

// Budget impact of pending invoice
after_approval = remaining - pending_invoice_amount
over_budget = after_approval < 0

// Prime cost %
prime_cost_pct = (total_food_bev_cogs + total_labor) / total_sales * 100

// SYN January 2026 example:
// food_bev_cogs = $5,307
// labor        = $19,053
// total_sales  = $13,213
// result       = (5307 + 19053) / 13213 * 100 = 184.4%
Section 05

Waste Log System

Status: Backlog — spec locked, build after the first 5 screens are working. Do not start here. Come back to this once core screens are validated.

Core Philosophy

Don't build a waste log. Build a waste moment. The operator is throwing something away. They should be able to log it in 5 seconds, on their phone, in the middle of the kitchen.

Design rule: If it can't feel like sending a photo in a text message, don't ship it.

The Capture Flow

1
Snap
Tap "Log Waste" → camera opens. Photograph the item being discarded.
2
Identify
NURA identifies from purchase history: "Strawberries — Baldor, last purchased Jan 29, $4.20/lb." Operator can override.
3
Estimate
Slider for rough quantity. Precision not required — "about 3 lbs" is enough for the intelligence layer to work.
4
Reason
One tap: Expired / Over-prepped / Quality issue / Customer return.
5
Logged
5 seconds total. Item, quantity, reason, estimated dollar value, timestamp — all saved automatically.

Waste Event Data Model

waste_events {
  id:                   uuid
  property_id:          uuid
  logged_by:            user_id
  logged_at:            timestamp
  item_name:            string        // "Strawberries"
  item_source_vendor:   string        // "Baldor"
  item_unit_cost:       decimal       // $4.20/lb
  quantity_estimated:   decimal       // 3.0
  quantity_unit:        string        // "lbs"
  estimated_dollar_value: decimal     // unit_cost * quantity
  reason:               enum(expired, over_prepped, quality, customer_return, other)
  reason_note:          string        // optional
  gl_code:              string        // inherited from purchase
  photo_url:            string        // optional
}

Status Signal Logic

ColorMeaningWhen to use
GreenHealthy / Under budgetBudget under 80%, costs on target, prime cost near 62%
AmberWatch / ElevatedBudget 80–100%, elevated but explainable, ramp-up pattern
OrangeAction neededOver budget, approval required, unusual spike
RedCriticalSignificantly over budget, audit variance, duplicate invoice

Key Component Rules

  • Big numbers: Newsreader font. Prime cost % should be large and dominant on its screen.
  • Progress bars: 8–12px height, rounded, green → amber at 80% → orange at 100%+
  • GL code pills: Small, subtle, surface-alt background. Secondary information — never the main event.
  • Status narrative: Always plain English. "Prime cost is elevated because labor is fixed while revenue ramps." Not just numbers.
  • Approval cards: Show 2×2 grid: Budget / After / Sales pace / Impact.
  • Mobile-first always: Operators use this on the floor. Large tap targets. Every action under 10 seconds.

  timezone:             string          -- "America/New_York"

users {
  id:                   uuid PRIMARY KEY
  email:                string UNIQUE
  role:                 enum(owner, gm, controller, viewer)

  created_at:           timestamp
}

gl_codes {
  id:                   uuid PRIMARY KEY
  property_id:          uuid REFERENCES properties
  code:                 string          -- "5217250"
  name:                 string          -- "Food Purchases"

vendors {
  id:                   uuid PRIMARY KEY
  property_id:          uuid REFERENCES properties
  name:                 string          -- "Baldor"
  default_gl_code:      string          -- "5217250"
  ap_email:             string
  delivery_frequency:   string          -- "weekly"
  is_active:            boolean DEFAULT true

invoices {
  id:                   uuid PRIMARY KEY
  property_id:          uuid REFERENCES properties
  vendor_id:            uuid REFERENCES vendors
  invoice_number:       string
  invoice_date:         date

Section 10

Design Principles You Must Not Break

1. "Clarity creates room for craft."

Financial visibility is a creative enabler, not a constraint. The product should feel like relief — like it gives operators more freedom to do what they love. Every screen should make the operator feel more in control, not more overwhelmed.

2. "The morning notification is the loyalty lock-in."

The daily habit is the product. If an operator opens NURA every morning without being prompted, you've won. Every decision should serve that daily check-in ritual.

3. "One action, five outcomes."

Never ask the operator to do something the system can do for them. One invoice upload triggers read, code, check, update, and surface insights — automatically.

4. "Month-end is confirmation, not discovery."

If an operator is surprised by something at month-end, NURA has failed. Everything important should be visible in real time so close is just clicking export on something already understood.

5. Never show a scary number without context.

184.4% prime cost without explanation looks catastrophic. With "labor is fixed, revenue is ramping, this is a ramp-up pattern" — it's understandable and manageable. Always pair the number with the narrative.

6. Mobile-first, always.

Operators live on the floor. Every interaction must be doable on a phone, with a thumb, in under 10 seconds.

7. "Suggest, don't ask."

Every question NURA asks should already contain the answer. The operator confirms. The system learns. Eventually NURA stops asking because it already knows.

8. "We win if the industry wins."

NURA is not extracting value from operators. It's helping them survive and thrive. The product should feel generous and entirely on the operator's side.
